Funktionen

Was ACI für Sie prüft

Eine statische Analyse, die Schwachstellen und Guideline-Verstöße findet, bevor der Code in Produktion geht.

SQL-Injection-Erkennung

Taint-Analyse verfolgt nicht vertrauenswürdige Eingaben durch Variablen und Routinen bis in dynamisches SQL – EXECUTE IMMEDIATE, EXECUTE und format() inklusive.

Coding-Guideline-Prüfung

Oracle-Regeln aus den Trivadis PL/SQL & SQL Coding Guidelines, für PostgreSQL ein eigenes Regelwerk – Namenskonventionen, Struktur und gute Praxis.

MITRE-ATT&CK-Indikatoren

Auffällige Muster werden Techniken des MITRE-ATT&CK-Frameworks zugeordnet – getrennte Regelsätze für Oracle und PostgreSQL.

Vier Report-Formate

Befunde als farbige Konsolenausgabe, als JSON, als eigenständiger HTML-Report oder als SARIF 2.1.0 für die Integration in Code-Scanning-Werkzeuge.

CI/CD-Profile

Fünf vorkonfigurierte Profile – advisory, ci, strict, audit und apex – steuern Exit-Code und Schweregradschwelle in der Pipeline.

Waiver & Regelintegrität

Inhaltsgebundene Fingerprints erlauben dokumentierte Ausnahmen über --waivers; mit --require-trusted-rules werden nur signierte Regeldateien akzeptiert.

Datenbanken

Zwei Plattformen, ein Werkzeug

ACI versteht die Dialekte beider Welten und prüft sie mit jeweils passenden Regelsätzen – rein statisch auf dem Quelltext.

Oracle

PL/SQL-Pakete, Prozeduren, Funktionen und Trigger.

  • Dynamisches SQL über EXECUTE IMMEDIATE und DBMS_SQL
  • Riskante Pakete wie UTL_FILE, UTL_HTTP und DBMS_*
  • DDL und Rechtevergabe (GRANT) im Laufzeitcode
  • Trivadis-Guidelines – Namen, Struktur und gute Praxis
  • Remote-Skripte über SQL*Plus @/@@/START (HTTP/FTP/UNC)

PostgreSQL

PL/pgSQL-Funktionen, Prozeduren und EDB-Erweiterungen.

  • Dynamisches SQL über EXECUTE und format()
  • Riskante Erweiterungen und COPY-Dateisystemzugriffe
  • DDL und GRANT in Funktionskörpern und Datenblöcken
  • Eigenes Guideline-Regelwerk speziell für PL/pgSQL
  • EPAS-Audit-Tampering und privilegierte SECURITY DEFINER-Routinen
Sicherheit

Fünf Sicherheitsprüfungen

Jeder Lauf durchsucht den Code nach diesen Schwachstellenklassen – ergänzt um inhaltsgebundene Fingerprints für reproduzierbare Befunde.

SQL-Injection

Verkettung nicht vertrauenswürdiger Werte in dynamisch ausgeführtes SQL – mit interprozeduraler Taint-Verfolgung und Verweisen auf die Quelle der Eingabe.

Unerwünschte Pakete

Aufrufe sicherheitskritischer Pakete und Erweiterungen – Datei-, Netzwerk- und Betriebssystemzugriffe, die in Datenbankcode selten gerechtfertigt sind.

Obfuskierter Code

Hex- und Base64-kodierte Zeichenketten sowie verschleierte Konstrukte, die Schadlogik vor einer Sichtprüfung verbergen sollen.

DDL im Code

CREATE, ALTER, DROP und GRANT innerhalb von Laufzeitlogik – häufig ein Hinweis auf Rechteausweitung.

Namenskonventionen

Bezeichner für Variablen, Parameter und Objekte werden gegen die Guideline-Konventionen geprüft – für einheitlichen, wartbaren Code.

Fingerprints & Waiver

Jeder Befund erhält einen inhaltsgebundenen Fingerprint – stabile Wiedererkennung über Läufe hinweg und Grundlage für dokumentierte Ausnahmen.

Funktionsweise

Vom Quelltext zum Report

ACI arbeitet rein statisch – ohne Datenbankverbindung, ohne Codeausführung. Eine schlanke Pipeline aus reinem Python (nur Standardbibliothek).

1

Quelltext

Oracle- oder PostgreSQL-Skripte werden eingelesen.

2

Lexer

Der Code wird in Token zerlegt – dialektbewusst.

3

Parser / IR

Routinen, Anweisungen und Ausdrücke als Zwischendarstellung.

4

Checks

Sicherheits- und Guideline-Prüfungen samt Taint-Analyse.

5

Report

Befunde als Konsole, JSON, HTML oder SARIF.

Konsole  farbig, sofort lesbar
JSON  maschinenlesbar
HTML  eigenständiger Report
