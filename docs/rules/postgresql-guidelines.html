PG-1010

SELECT * vermeiden

Minor

SELECT * liest alle Spalten und ist bei Schemaaenderungen fehleranfaellig.

MerkmalMaintainability
DetektorTyp regex · Muster \bSELECT\s+\* · Ziel masked
EmpfehlungDie benoetigten Spalten explizit aufzaehlen.
Verstoß
SELECT * INTO v_emp FROM employees WHERE id = p_id;
Konform
SELECT id, name, dept_id
  INTO v_id, v_name, v_dept
  FROM employees WHERE id = p_id;
PG-1020

Vergleich mit = NULL statt IS NULL

Major

Ein Vergleich '= NULL' ergibt in SQL immer NULL (unbekannt) - gemeint ist fast immer 'IS NULL'.

MerkmalReliability
DetektorTyp regex · Muster (?<![<>!:])=\s*NULL\b · Ziel masked
EmpfehlungIS NULL bzw. IS NOT NULL verwenden.
Verstoß
-- '= NULL' ergibt immer NULL - der Zweig wird nie ausgefuehrt
IF v_name = NULL THEN
  handle_missing();
END IF;
Konform
IF v_name IS NULL THEN
  handle_missing();
END IF;
PG-1021

Vergleich mit <> NULL bzw. != NULL

Major

Ein Vergleich '<> NULL' / '!= NULL' liefert immer NULL - gemeint ist 'IS NOT NULL'.

MerkmalReliability
DetektorTyp regex · Muster (?:<>|!=)\s*NULL\b · Ziel masked
EmpfehlungIS NOT NULL verwenden.
Verstoß
-- '<> NULL' liefert immer NULL statt TRUE/FALSE
IF v_status <> NULL THEN
  process();
END IF;
Konform
IF v_status IS NOT NULL THEN
  process();
END IF;
PG-1070

Verschachtelte Blockkommentare

Minor

Verschachtelte /* ... */-Kommentare sind schwer lesbar und fehleranfaellig.

MerkmalMaintainability
DetektorTyp builtin · Detektor nested_comments
EmpfehlungBlockkommentare nicht verschachteln.
Verstoß
/* aeusserer Kommentar
   /* innerer Block */   -- das aeussere */ beendet hier vorzeitig
   v_x := 1;
*/
Konform
-- Zum Auskommentieren Zeilenkommentare verwenden
-- v_x := 1;
-- v_y := 2;
PG-2010

varchar ohne Laengenangabe

Info

varchar / character varying ohne Laenge entspricht 'text' - die Absicht bleibt unklar.

MerkmalMaintainability
DetektorTyp regex · Muster \b(?:varchar|character\s+varying)\b(?!\s*\() · Ziel masked
EmpfehlungBewusst 'text' verwenden oder eine Laenge angeben.
Verstoß
CREATE TABLE app_user (
  id    integer,
  name  varchar          -- ohne Laenge = text, Absicht unklar
);
Konform
CREATE TABLE app_user (
  id    integer,
  name  varchar(100)     -- feste Laenge, oder bewusst 'text'
);
PG-3010

COUNT(*) zur Existenzpruefung

Minor

COUNT(*) > 0 zaehlt unnoetig alle Zeilen - EXISTS bricht beim ersten Treffer ab.

MerkmalEfficiency
DetektorTyp regex · Muster \bcount\s*\(\s*\*\s*\)\s*(?:>\s*0|>=\s*1) · Ziel masked
EmpfehlungEXISTS (SELECT 1 ...) statt COUNT(*) > 0 verwenden.
Verstoß
-- zaehlt unnoetig alle Zeilen
SELECT count(*) > 0 INTO v_has_rows
  FROM orders WHERE customer_id = p_id;
Konform
-- EXISTS bricht beim ersten Treffer ab
SELECT EXISTS (
  SELECT 1 FROM orders WHERE customer_id = p_id
) INTO v_has_rows;
PG-3020

NOT IN mit Unterabfrage

Major

NOT IN (SELECT ...) liefert ueberraschend keine Zeilen, sobald die Unterabfrage einen NULL-Wert enthaelt.

MerkmalReliability
DetektorTyp regex · Muster \bNOT\s+IN\s*\(\s*SELECT\b · Ziel masked
EmpfehlungNOT EXISTS oder einen Anti-Join (LEFT JOIN ... IS NULL) verwenden.
Verstoß
-- liefert keine Zeilen, sobald die Unterabfrage ein NULL enthaelt
SELECT id INTO v_id FROM customers
 WHERE id NOT IN (SELECT customer_id FROM orders);
Konform
-- NOT EXISTS ist NULL-sicher
SELECT id INTO v_id FROM customers c
 WHERE NOT EXISTS (
   SELECT 1 FROM orders o WHERE o.customer_id = c.id);
PG-3310

COMMIT/ROLLBACK in einer Schleife

Major

Transaktionssteuerung innerhalb einer Schleife ist meist unbeabsichtigt und teuer.

MerkmalReliability
DetektorTyp builtin · Detektor commit_in_loop
EmpfehlungTransaktionsgrenzen ausserhalb der Schleife setzen.
Verstoß
FOR r IN SELECT id FROM staging LOOP
  PERFORM process(r.id);
  COMMIT;              -- COMMIT je Iteration
END LOOP;
Konform
FOR r IN SELECT id FROM staging LOOP
  PERFORM process(r.id);
END LOOP;
COMMIT;                -- ein COMMIT nach der Schleife
PG-4210

Mehrere ELSIF-Zweige

Minor

Eine IF-Anweisung mit mehreren ELSIF-Zweigen ist als CASE oft lesbarer.

MerkmalMaintainability
DetektorTyp builtin · Detektor case_for_elsif
EmpfehlungMehrfachverzweigungen als CASE formulieren.
Verstoß
IF v_grade = 'A' THEN v_bonus := 100;
ELSIF v_grade = 'B' THEN v_bonus := 50;
ELSIF v_grade = 'C' THEN v_bonus := 20;
ELSE v_bonus := 0;
END IF;
Konform
v_bonus := CASE v_grade
             WHEN 'A' THEN 100
             WHEN 'B' THEN 50
             WHEN 'C' THEN 20
             ELSE 0
           END;
PG-5010

Leerer WHEN-OTHERS-Handler

Major

WHEN OTHERS THEN NULL verschluckt jede Ausnahme spurlos und versteckt Fehler.

MerkmalReliability
DetektorTyp regex · Muster \bWHEN\s+OTHERS\s+THEN\s+NULL\s*; · Ziel masked
EmpfehlungAusnahme protokollieren und/oder mit RAISE erneut werfen.
Verstoß
BEGIN
  PERFORM process();
EXCEPTION
  WHEN OTHERS THEN NULL;   -- Ausnahme wird spurlos verschluckt
END;
Konform
BEGIN
  PERFORM process();
EXCEPTION
  WHEN OTHERS THEN
    RAISE WARNING 'process fehlgeschlagen: %', SQLERRM;
    RAISE;
END;
PG-7010

SECURITY DEFINER ohne festen search_path

Major

Eine SECURITY-DEFINER-Funktion ohne 'SET search_path' ist anfaellig fuer search_path-Angriffe.

MerkmalSecurity
DetektorTyp builtin · Detektor security_definer_search_path
EmpfehlungBei SECURITY DEFINER stets einen festen 'SET search_path = ...' setzen.
Verstoß
-- SECURITY DEFINER ohne SET search_path -> search_path-Angriff moeglich
CREATE FUNCTION get_balance(p_id int) RETURNS numeric
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
BEGIN
  RETURN (SELECT balance FROM app.accounts WHERE id = p_id);
END;
$$;
Konform
CREATE FUNCTION get_balance(p_id int) RETURNS numeric
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = app, pg_temp
AS $$
BEGIN
  RETURN (SELECT balance FROM app.accounts WHERE id = p_id);
END;
$$;
PG-7011

SECURITY DEFINER mit riskantem search_path

Major

SECURITY-DEFINER-Funktion mit riskantem search_path (z.B. public oder pg_temp nicht am Ende).

MerkmalSecurity
DetektorTyp builtin · Detektor security_definer_unsafe_search_path
EmpfehlungEinen festen, minimalen search_path setzen; public vermeiden und pg_temp ans Ende stellen.
Verstoß
-- 'public' im search_path einer DEFINER-Funktion ist angreifbar
CREATE FUNCTION get_balance(p_id int) RETURNS numeric
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public, app
AS $$ BEGIN RETURN app.balance_of(p_id); END; $$;
Konform
-- minimaler search_path, pg_temp am Ende, kein 'public'
CREATE FUNCTION get_balance(p_id int) RETURNS numeric
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = app, pg_temp
AS $$ BEGIN RETURN app.balance_of(p_id); END; $$;
PG-7012

SECURITY DEFINER mit dynamischem EXECUTE

Major

SECURITY-DEFINER-Funktion fuehrt dynamisches SQL aus.

MerkmalSecurity
DetektorTyp builtin · Detektor security_definer_dynamic_execute
EmpfehlungDynamisches SQL in SECURITY-DEFINER-Funktionen vermeiden oder besonders streng validieren und qualifizieren.
Verstoß
CREATE FUNCTION wipe(p_tab text) RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = app, pg_temp
AS $$
BEGIN
  EXECUTE 'TRUNCATE ' || p_tab;   -- dynamisches SQL mit DEFINER-Rechten
END;
$$;
Konform
-- Festes, statisches SQL statt dynamischem EXECUTE
CREATE FUNCTION clear_staging() RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = app, pg_temp
AS $$
BEGIN
  TRUNCATE app.staging_rows;
END;
$$;
PG-7013

SECURITY DEFINER mit unqualifiziertem Funktionsaufruf

Major

SECURITY-DEFINER-Funktion enthaelt einen unqualifizierten Funktionsaufruf.

MerkmalSecurity
DetektorTyp builtin · Detektor security_definer_unqualified_call
EmpfehlungFunktionsaufrufe in SECURITY-DEFINER-Code schemaqualifiziert ausfuehren oder search_path strikt absichern.
Verstoß
CREATE FUNCTION post(p_id int) RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = app, pg_temp
AS $$
BEGIN
  PERFORM calc_total(p_id);      -- unqualifiziert: angreifbar
END;
$$;
Konform
CREATE FUNCTION post(p_id int) RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = app, pg_temp
AS $$
BEGIN
  PERFORM app.calc_total(p_id);  -- schemaqualifiziert
END;
$$;

PG-1080

Gleicher Ausdruck auf beiden Seiten eines Vergleichs

Major

Auf beiden Seiten eines Vergleichs steht derselbe Bezeichner (z.B. 'x = x') - das ist fast immer ein Logikfehler oder Tippfehler.

MerkmalReliability
DetektorTyp regex · Muster \b([A-Za-z_][A-Za-z0-9_$]*)\s*(?:<>|!=|<=|>=|=|<|>)\s*\1\b · Ziel masked
EmpfehlungVergleich pruefen; vermutlich war ein anderer Operand oder 'IS NULL' gemeint.
PG-3110

Zielspalten bei INSERT immer angeben

Minor

INSERT ... VALUES ohne explizite Spaltenliste ist bei Schemaaenderungen fehleranfaellig (Reihenfolge/Anzahl der Spalten).

MerkmalMaintainability
DetektorTyp regex · Muster \bINSERT\s+INTO\s+(?:"[^"]+"|[A-Za-z_][\w$.]*)\s+VALUES\b · Ziel masked
EmpfehlungDie Zielspalten explizit auffuehren: INSERT INTO t (a, b) VALUES (...).
PG-3190

NATURAL JOIN vermeiden

Minor

NATURAL JOIN verknuepft ueber alle gleichnamigen Spalten - eine spaetere Spaltenaenderung kann die Join-Bedingung still veraendern.

MerkmalReliability
DetektorTyp regex · Muster \bNATURAL\s+(?:(?:INNER|LEFT|RIGHT|FULL)\s+)?(?:OUTER\s+)?JOIN\b · Ziel masked
EmpfehlungJoin-Bedingung explizit mit ON bzw. USING angeben.
PG-4270

Boolean-Werte nicht mit Boolean-Literalen vergleichen

Minor

Ein Vergleich mit TRUE/FALSE (z.B. 'flag = TRUE') ist redundant - der Boolean-Ausdruck kann direkt verwendet werden.

MerkmalMaintainability
DetektorTyp regex · Muster (?:(?<![:<>!])=|<>|!=)\s*(?:TRUE|FALSE)\b|\b(?:TRUE|FALSE)\s*(?:(?<![:<>!])=|<>|!=) · Ziel masked
EmpfehlungStatt 'x = TRUE' einfach 'x' und statt 'x = FALSE' 'NOT x' schreiben.
PG-7125

CREATE OR REPLACE statt CREATE allein verwenden

Info

CREATE FUNCTION/PROCEDURE/VIEW ohne OR REPLACE schlaegt fehl, wenn das Objekt bereits existiert - erschwert idempotente Deployments.

MerkmalMaintainability
DetektorTyp regex · Muster \bCREATE\s+(?!OR\s+REPLACE\b)(?:TEMP\s+|TEMPORARY\s+)?(?:FUNCTION|PROCEDURE|VIEW)\b · Ziel masked
EmpfehlungCREATE OR REPLACE FUNCTION/PROCEDURE/VIEW verwenden.
PG-2210

numeric/decimal ohne Genauigkeit deklarieren

Infoinaktiv

numeric/decimal ohne (Genauigkeit, Skala) erlaubt beliebige Praezision - oft unbeabsichtigt. In PostgreSQL ist unbeschraenktes numeric jedoch legitim, daher standardmaessig deaktiviert.

MerkmalMaintainability
DetektorTyp regex · Muster \b(?:numeric|decimal)\b(?!\s*\() · Ziel masked
EmpfehlungGenauigkeit/Skala angeben (numeric(p, s)) - oder bewusst unbeschraenktes numeric belassen.
PG-2310

Datentyp char(n) vermeiden (text bevorzugen)

Infoinaktiv

char(n) fuellt Werte mit Leerzeichen auf (Blank Padding) und bietet in PostgreSQL keinen Performance-Vorteil gegenueber text/varchar.

MerkmalMaintainability
DetektorTyp regex · Muster (?<![A-Za-z0-9_$])char\s*\( · Ziel masked
Empfehlungtext (oder varchar mit Laenge) statt char(n) verwenden.
PG-2330

Leere Zeichenkette nicht wie NULL behandeln (Oracle-Migrationsfalle)

Majorinaktiv

Anders als Oracle behandelt PostgreSQL die leere Zeichenkette '' NICHT als NULL ('' IS NOT NULL). Aus Oracle migrierter Code, der sich auf ''=NULL verlaesst, verhaelt sich in PostgreSQL anders.

MerkmalReliability
EmpfehlungAuf '' und NULL getrennt pruefen; NULL-Pruefungen explizit mit IS NULL / IS NOT NULL durchfuehren.
