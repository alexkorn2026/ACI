APEX / ORDS ACI-APEX-ORDS

ACI-eigene heuristische Regeln fuer Oracle APEX und ORDS: benutzerkontrollierte Page Items / Session State in dynamischem SQL, SSRF/XSS ueber APEX-APIs, AutoREST-Exposition und schwache Autorisierung. Bewusst heuristisch, hohe Signalstaerke. Kein vollstaendiger APEX-/ORDS-Parser.

7 Regeln

ACI-APEX-DYNSQL-SESSION-STATE

SQL aus APEX Session State per Konkatenation gebaut

High

SQL-Text wird per ||-Konkatenation aus einem APEX Page Item / Session State (:P1_x, V('P1_x'), APEX_UTIL.GET_SESSION_STATE) gebaut und (ohne EXECUTE-Sink) zurueckgegeben - typisch fuer 'PL/SQL Function Body returning SQL', Region Sources und Reports. Benutzerkontrollierte Werte fliessen ungeprueft in den SQL-Text: SQL-Injection.

MITRE-TechnikSQL Injection via APEX Session State (Function Body returning SQL)
DetektorTyp builtin · Detektor apex_concat_sql
EmpfehlungWerte als Bindevariablen IM SQL-Text uebergeben (z.B. 'where id = :P1_ID') statt sie zu konkatenieren. Objektnamen mit DBMS_ASSERT absichern.
Quellehttps://owasp.org/www-community/attacks/SQL_Injection
ACI-APEX-EXPORT-DYNSQL

APEX-Export: SQL aus Session State in einem wwv_flow_api-Argument

High

Ein code-tragendes APEX-Export-Argument (z.B. p_plug_source, p_query, p_function_body) baut SQL per ||-Konkatenation aus einem APEX Page Item / Session State. In exportierten Apps ist das die typische 'PL/SQL Function Body returning SQL'-Schwachstelle - der Code steht hier als String-Literal und entgeht den regulaeren Checks.

MITRE-TechnikSQL Injection via APEX Session State (Export-Datei)
DetektorTyp builtin · Detektor apex_export_code_sql
EmpfehlungWerte als Bindevariablen IM SQL-Text uebergeben (:P1_x) statt sie zu konkatenieren; Objektnamen mit DBMS_ASSERT absichern. Befund moeglichst in der APEX-Anwendung selbst korrigieren.
Quellehttps://owasp.org/www-community/attacks/SQL_Injection

MITRE-TechnikCommand and Scripting Interpreter (SQL*Plus HOST)
DetektorTyp builtin · Detektor client_directive

ACI-ORA-SQLPLUS-REMOTE-SCRIPT

Ausfuehrung eines Remote-Skripts ueber @ / @@ / START (URL bzw. UNC-Pfad)

Critical

SQL*Plus-/edbplus-Direktive @, @@ oder START laedt und fuehrt ein Skript von einem entfernten Ort aus (http://, ftp:// oder UNC-Netzwerkpfad). Der Skriptpfad ist vollstaendig fremdbestimmt und wird ueber einen nicht authentifizierten Klartext-Transport bezogen - beliebiges Remote-SQL/PL-SQL laeuft mit den Rechten des ausfuehrenden Benutzers. (Hinweis: Oracle fuehrt http:// und ftp:// aus; https:// wird offiziell nicht unterstuetzt, der Aufruf gilt aber als verdaechtig.)

MITRE-TechnikIngress Tool Transfer / Remote Script Execution (SQL*Plus @ @@ START)
DetektorTyp builtin · Detektor client_directive
EmpfehlungNiemals Skripte ueber URL oder Netzwerkpfad einbinden. Nur lokale, vertrauenswuerdige, versionierte Skripte mit festem Pfad aufrufen; den Remote-Bezug von Skripten in Deployment-Prozessen unterbinden.
Quellehttps://attack.mitre.org/techniques/T1105/

SQL*Plus-/edbplus-Direktive @, @@ oder START fuehrt ein externes SQL-Skript aus. In Deployment-Skripten ein Einfallstor, wenn der Skriptpfad nicht kontrolliert ist (eingebundenes Skript laeuft mit den Rechten des ausfuehrenden Benutzers).

MITRE-TechnikCommand and Scripting Interpreter (SQL*Plus/edbplus Skriptaufruf)
DetektorTyp builtin · Detektor client_directive
EmpfehlungNur vertrauenswuerdige, versionierte Skripte einbinden; Skriptpfade nicht aus Variablen/Eingaben zusammensetzen.
Quellehttps://attack.mitre.org/techniques/T1059/

MITRE-TechnikCommand and Scripting Interpreter (variabler Skriptpfad)
DetektorTyp builtin · Detektor client_directive

ACI-ORA-SQLPLUS-SUBSTITUTION

SQL*Plus-/edbplus-Substitutionsvariable (& / &&)

Warning

Substitutionsvariable in einer sicherheitsrelevanten Anweisung (CREATE/ALTER/DROP/GRANT/REVOKE/EXECUTE IMMEDIATE/IDENTIFIED BY). Der Wert wird vom Client VOR der Ausfuehrung rein textuell ersetzt - ohne Escaping ein SQL-Injection-Vektor, wenn er aus ungeprueften Quellen stammt. MITRE-TechnikCommand and Scripting Interpreter (Substitutionsvariable)
DetektorTyp builtin · Detektor sqlplus_security_substitution

SQL*Plus-/edbplus-Direktive ACCEPT bzw. DEFINE definiert eine Substitutionsvariable (haeufig aus interaktiver Eingabe). In Verbindung mit & wird daraus eine ungeprueft eingesetzte Eingabe.

ACI-ORA-SQLPLUS-WHENEVER-CONTINUE

WHENEVER SQLERROR/OSERROR CONTINUE - Fehler werden ignoriert

Warning

WHENEVER SQLERROR/OSERROR CONTINUE laesst das Skript nach einem Fehler weiterlaufen. In Deployment-/Migrationsskripten koennen so fehlgeschlagene sicherheitsrelevante Schritte (z.B. ein REVOKE oder eine Rechtevergabe) unbemerkt uebergangen werden - das Skript endet scheinbar erfolgreich in einem inkonsistenten Zustand.

MITRE-TechnikRobuste Fehlerbehandlung (Fehler werden ignoriert)
DetektorTyp builtin · Detektor client_directive
EmpfehlungIn Deployment-Skripten WHENEVER SQLERROR EXIT FAILURE (und WHENEVER OSERROR EXIT) setzen, damit Fehler den Lauf abbrechen.
Quellehttps://attack.mitre.org/techniques/T1059/

-- Unified Auditing per Policy erfasst, nicht über eigenen Code.
NULL;

T1548

Programmeinheit mit AUTHID DEFINER

Warning

AUTHID DEFINER - die Einheit läuft mit den Rechten des Eigentümers. In Kombination mit dynamischem SQL oder DDL ein klassischer Pfad zur Rechteausweitung.

MITRE-TechnikAbuse Elevation Control Mechanism (Definer Rights)
DetektorTyp regex · Muster \bAUTHID\s+DEFINER\b · Ziel code
EmpfehlungPrüfen, ob DEFINER-Rechte nötig sind. Privilegierte DEFINER-Einheiten, die Parameter in dynamisches SQL übernehmen, mit DBMS_ASSERT absichern oder auf AUTHID CURRENT_USER umstellen.
Quellehttps://attack.mitre.org/techniques/T1548/
Verwundbar
-- DEFINER-Rechte + ungeprüftes dynamisches SQL = Rechteausweitung
CREATE OR REPLACE PROCEDURE run_q(p_tab VARCHAR2)
  AUTHID DEFINER AS
BEGIN
  EXECUTE IMMEDIATE 'SELECT count(*) FROM ' || p_tab;
END;
Sicher
-- AUTHID CURRENT_USER: Ausführung mit den Rechten des Aufrufers;
-- Bezeichner zusätzlich über DBMS_ASSERT prüfen.
CREATE OR REPLACE PROCEDURE run_q(p_tab VARCHAR2)
  AUTHID CURRENT_USER AS
BEGIN
  EXECUTE IMMEDIATE
    'SELECT count(*) FROM ' || DBMS_ASSERT.SQL_OBJECT_NAME(p_tab);
END;
