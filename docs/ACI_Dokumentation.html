Kritische Einschränkungen

Diese Hinweise stehen bewusst am Anfang der Dokumentation, weil sie für eine Audit- bzw. Freigabe-Bewertung von ACI maßgeblich sind.

ACI ist kein vollständiger SQL-/PL/SQL-/PL/pgSQL-Compiler. Es bleibt ein heuristischer statischer Scanner. Ein Auditor sollte deshalb folgende Grenzen akzeptieren:

  • False Positives und False Negatives sind möglich. ACI ersetzt kein manuelles Review.
  • Dynamisch zusammengesetztes SQL wird gut, aber nicht formal vollständig analysiert. False Negatives bleiben insbesondere bei SQL möglich, das ACI nicht statisch vorliegt – dynamisches SQL über mehrere Dateien hinweg, Build-Templates, Code-/Migrations-Generatoren oder konfigurations-/datengetriebenes SQL (siehe Abschnitt 13. Grenzen & Hinweise).
  • Regeldateien sind Teil der Trust Boundary: Wer die Regeln (aci/rules/, --rules/--rules-dir) manipulieren kann, kann den CI/CD-Gate schwächen.
  • PostgreSQL-kritische CI/CD-Policies liegen teils in MITRE- bzw. Spezialdetektoren und nicht einheitlich in ddl_in_code.critical_statements – funktional in Ordnung, governance-seitig aber etwas verstreut.
  • High ist für CI/CD praktisch oft genauso blockierwürdig wie Critical: für eine echte Deployment-Blockade unsicherer Befehle daher --fail-on high verwenden (siehe Abschnitt 11. CI/CD-Integration).

Eine ausführlichere Darstellung enthält Abschnitt 13. Grenzen & Hinweise.

Developer Onboarding

Lies diesen Abschnitt zuerst. Er ist der zentrale Einstieg für Entwickler, die ACI übernehmen oder weiterentwickeln. Er gibt den Überblick und navigiert zu den Detailabschnitten – er dupliziert sie nicht.

High-Level-Pipeline

ACI verarbeitet jede Datei in einer festen Kette; jede Stufe entspricht einem Modul in aci/:

Quelltext
  -> Lexer            aci/lexer.py    Tokens, Maskierung von Kommentaren/Strings
  -> Parser / IR      aci/parser.py   leichte IR: Routinen, Statements, dyn. SQL
                      aci/ir.py       IR-Datenmodell + line_starts()
  -> Source-Fassade   aci/source.py   buendelt Lexer + IR pro Datei
  -> Checks           aci/checks/     Security-Checks, Guideline-/MITRE-Detektoren
  -> Findings         aci/finding.py  Finding, Severity, Fingerprint
  -> Waiver           aci/waivers.py  Ausnahmen anwenden (nicht entfernen)
  -> Integritaet      aci/integrity.py Ruleset-Hash + Trust-Status
  -> Reporter         aci/reporting/  console / json / html / sarif
  -> CLI              aci/cli.py      _run() orchestriert die Kette

Einstiegspunkt ist aci/cli.py:_run(); die Funktion ist bewusst in kleine Teilschritte zerlegt (Regeln laden, Optionen prüfen, Integrität verifizieren, scannen, Waiver anwenden, Reports schreiben, Exit-Code bestimmen). Wer den Ablauf verstehen will, liest zuerst _run().

Lokales Setup

python3 --version            # Python 3.9+ erforderlich
python3 -m pytest -q          # Testsuite (keine Laufzeitabhaengigkeiten)
python3 -m compileall -q aci  # Syntaxpruefung
python3 -m ruff check .       # Linter   (falls installiert)
python3 -m mypy aci           # Typpruefung (falls installiert)
python3 -m aci samples/vulnerable_oracle.sql -g security

ACI nutzt ausschließlich die Standardbibliothek; ruff und mypy sind reine Entwickler-Werkzeuge (pip install .[dev]).

Kernkonzepte

  • IR (Intermediate Representation) – eine bewusst leichtgewichtige Zwischenform, kein vollständiger SQL-Compiler. Sie bewahrt Statement- und Routinegrenzen sowie die Reihenfolge von Zuweisungen (siehe 3. Architektur).
  • Maskierung – code_masked ersetzt Kommentar- und String-Inhalte, sodass Regex-Detektoren nicht auf Daten statt Code anschlagen.
  • Taint-Quelle – bei SQL-Injection wird neben der Fundstelle (Sink) die Herkunft des dynamischen Strings ausgewiesen; Details und alle Begriffe im 14. Glossar.
  • Regeln sind Daten – Checks lesen JSON aus aci/rules/; neue Regeln brauchen i. d. R. keinen Code (siehe 10. Regeldateien).

Dialekt-Fallen

  • Der Dialekt wird nicht erraten: ohne --dialect postgresql gilt der Oracle-Regelsatz.
  • Oracle- und PostgreSQL-Regeln liegen getrennt (oracle.json / postgresql.json, Unterverzeichnisse oracle/ / postgresql/).
  • PostgreSQL-Dollar-Quotes müssen verschachtelt unterschiedliche Tags verwenden ($func$ / $msg$).

Typische Stolpersteine

  • Regex-Detektoren für mehrteilige Anweisungen werden statement-lokal ausgewertet (Auswertung je ;-Region) – ein [\s\S]*? darf eine Statement-Grenze nicht überschreiten, sonst entstehen False Positives über mehrere Statements hinweg.
  • Die Datenflussanalyse ist routine-lokal; sie folgt keinen Aufrufen über Routinegrenzen hinweg.
  • Reports können Quelltext enthalten – für CI-Artefakte --safe-report verwenden.
  • Findings und Reporttexte sind teils deutschsprachig und werden von Tests geprüft: Wortlaut nicht unbedacht ändern.

Tiefe folgt in den verlinkten Abschnitten; im Fehlerfall hilft 14. Debugging.

1. Überblick

ACI (Automated Code Inspection) ist ein statisches Analyse-Werkzeug, das übergebenen Datenbank-Code – Oracle PL/SQL oder PostgreSQL PL/pgSQL – untersucht. ACI führt den Code nicht aus, sondern prüft ausschließlich den Quelltext.

ACI prüft in zwei Gruppen:

  • Sicherheit – fünf Sicherheits-Checks (SQL-Injection, unerwünschte Packages, obfuskierter Code u.a.). Schweregrade: Warning High Critical
  • Coding Guidelines – Prüfungen nach den Trivadis PL/SQL & SQL Coding Guidelines (Stil, Wartbarkeit, Zuverlässigkeit). Schweregrade: Info Minor Major Critical Blocker. Diese Gruppe steht nur für Oracle zur Verfügung.

Beide Gruppen werden im Report getrennt dargestellt, jede mit ihrer eigenen Schweregrad-Skala. Die Regeln werden in externen JSON-Dateien gepflegt und sind ohne Code-Änderung erweiterbar.

Hauptmerkmale

  • Zwei Prüfgruppen mit getrennten Schweregrad-Skalen
  • Fünf Sicherheits-Checks plus MITRE-ATT&CK- und APEX-/ORDS-Regeln sowie über 50 aktive Coding-Guideline-Regeln
  • Heuristische Erkennung für Client-/Deployment-Skripte (SQL*Plus, edbplus, psql) und Oracle APEX/ORDS (inkl. APEX-Export-Dateien)
  • Zwei SQL-Dialekte: Oracle und PostgreSQL (EPAS-bewusst)
  • Vier Ausgabeformate: Konsole, JSON, HTML, SARIF
  • Externe, frei editierbare JSON-Regeldateien
  • Einbindbar in CI/CD über Exit-Codes
  • Reine Python-Standardbibliothek – keine Abhängigkeiten

2. Funktionsweise

Ein Scan durchläuft für jede Datei vier Schritte:

  1. Regeln laden – die Sicherheits-Regeldatei des Dialekts und (für Oracle) die Coding-Guideline-Kategoriedateien werden eingelesen und validiert.
  2. Vorverarbeitung – der Quelltext wird einmalig aufbereitet: Kommentare werden entfernt und String-Literale erkannt. Dadurch lösen Schlüsselworte in Kommentaren keine Fehlalarme aus, und Checks können gezielt im Code oder im String-Inhalt suchen.
  3. Checks ausführen – alle aktiven Checks beider Gruppen werden ausgeführt und liefern Findings.
  4. Bericht erstellen – die Findings werden nach Gruppe getrennt zusammengeführt, sortiert und in den gewählten Formaten ausgegeben.
Die Vorverarbeitung erzeugt zwei aufbereitete Textvarianten: eine ohne Kommentare (String-Literale bleiben erhalten) und eine zusätzlich maskierte Variante, in der auch der Inhalt von String-Literalen ausgeblendet ist. Beide Varianten sind zeichengenau gleich lang wie das Original, sodass Fundstellen exakt auf Zeile und Spalte abgebildet werden.

3. Architektur & Aufbau

ACI ist als Python-Paket organisiert. Jeder Sicherheits-Check ist eine Klasse; jede Coding-Guideline-Regel wird durch einen GuidelineCheck mit Detektor-Beschreibung ausgeführt.

ACI/
│
├─ aci.py                  Kommandozeilen-Einstiegspunkt
├─ aci/                    Python-Paket
│   ├─ cli.py              CLI - _run() orchestriert die Pipeline
│   ├─ config.py           Standardparameter aus aci.ini
│   ├─ finding.py          Datenmodell: Finding, Severity, Fingerprint
│   ├─ lexer.py            Kommentare/Strings maskieren
│   ├─ parser.py / ir.py   leichtgewichtige Parser-/IR-Schicht
│   ├─ source.py           Vorverarbeitung je Datei
│   ├─ rules.py            Laden der Sicherheits-/Guideline-/MITRE-Regeln
│   ├─ scanner.py          Scan-Engine (Dateien/Verzeichnisse)
│   ├─ waivers.py          Waiver-/Ausnahmeprozess
│   ├─ integrity.py        Ruleset-Hash + Vertrauensstatus
│   ├─ checks/             Pruef-Checks (Paket)
│   │   ├─ base.py        Check-Basisklasse + gemeinsame Helfer
│   │   ├─ lexical.py     Naming-, Packages-, Obfuscation-Check
│   │   ├─ sqli.py        SQL-Injection-Check (Datenfluss)
│   │   ├─ ddl.py         DDL-im-Code-Check (Datenfluss)
│   │   ├─ detectors.py   eingebaute Guideline-/MITRE-Detektoren
│   │   └─ guidelines.py  GuidelineCheck, MitreCheck
│   └─ reporting/          Reportformate (Paket)
│       ├─ report.py      ScanReport (Datenmodell)
│       ├─ console.py     Konsolen-Report
│       ├─ json_report.py JSON-Report
│       ├─ sarif.py       SARIF-Report
│       └─ html.py        HTML-Report
├─ rules/                  externe Regeldateien
│   ├─ oracle.json             Sicherheit (Oracle)
│   ├─ postgresql.json         Sicherheit (PostgreSQL)
│   └─ guidelines/oracle/      Coding Guidelines (je Kategorie)
├─ samples/                Beispieldateien
└─ docs/                   diese Dokumentation

Die beiden umfangreichsten Bereiche sind als Unterpakete organisiert: aci/checks/ trennt Basisklasse, lexikalische Checks, Datenfluss-Checks (SQL-Injection/DDL), Detektor-Bibliothek und GuidelineCheck; aci/reporting/ trennt das Datenmodell von je einem Modul pro Reportformat. Beide besitzen ein re-exportierendes __init__.py, sodass from aci.checks import … bzw. from aci.reporting import … unverändert funktioniert.

Wichtig: Die Coding Guidelines werden in einer JSON-Datei je Trivadis-Kategorie gepflegt (gute Wartbarkeit), erscheinen im Report jedoch als eine gemeinsame Gruppe.

Warum kein Compiler – bewusst ein heuristischer Scanner

ACI ist kein vollständiger SQL-/PL/SQL-/PL/pgSQL- Compiler, sondern bleibt bewusst ein heuristischer statischer Scanner. Das ist eine Architekturentscheidung, keine Unfertigkeit:

  • Ein Compiler löst Probleme prozeduralen Codes – vollständige Daten- und Kontrollflussanalyse, Sichtbarkeits- und Typauflösung. Diesen Aufwand bräuchte man für Stored Procedures, Funktionen und Trigger. Reine DDL (CREATE TABLE, VIEW, SEQUENCE, GRANT) ist dagegen deklarativ: kein Datenfluss, kein Kontrollfluss. Sie gut zu prüfen heißt, die einzelne Anweisung sauber abzugrenzen und gegen Regeln zu halten – dafür ist kein Compiler nötig.
  • Echte semantische Prüfung bräuchte den Datenbank-Katalog. Fragen wie „existiert diese Spalte, stimmt der Typ?“ lassen sich nur gegen ein laufendes Schema beantworten. ACI ist bewusst ein offline, DB-loses Werkzeug, das nur den übergebenen Quelltext liest – ein Compiler ohne Katalog brächte diese Stufe ohnehin nicht.
  • Die leichtgewichtige Parser-/IR-Schicht reicht gezielt so weit, wie sie Präzision bringt – Statement- und Routine-Grenzen, dynamisches SQL, Reihenfolge von Zuweisungen – und bewusst nicht weiter. Sie gibt nicht vor, eine vollständige SQL-Grammatik oder Kontrollfluss-Engine zu sein.
  • Die Entscheidung bringt konkrete Vorteile: nur Python-Standardbibliothek (keine Parser-Abhängigkeit, kleine Lieferkette), robust über Dialekt- und Versionsunterschiede hinweg, schnell und wartbar.

Die Kehrseite wird offen benannt: Eine Heuristik kann False Positives und False Negatives erzeugen. ACI ist daher ein Review-Assistent und CI-Signal, kein formaler Sicherheitsbeweis – siehe Abschnitt 13. Grenzen & Hinweise.

6. Prüfgruppen & Schweregrade

Jedes Finding gehört zu genau einer Gruppe und trägt einen Schweregrad aus der Skala dieser Gruppe.

Gruppe Sicherheit

LevelBedeutung
Warning Auffälligkeit / Verstoß gegen gute Praxis
High ernstzunehmendes Risiko, sollte behoben werden
Critical schwerwiegende Lücke mit hohem Missbrauchspotenzial

Gruppe Coding Guidelines

Diese Gruppe nutzt die fünfstufige Skala der Trivadis-Guidelines:

LevelBedeutung
Info Hinweis ohne unmittelbare Auswirkung
Minor geringfügiger Verstoß (Stil, Lesbarkeit)
Major deutlicher Verstoß gegen Wartbarkeit/Zuverlässigkeit
Critical schwerwiegender Verstoß
Blocker Verstoß, der die Auslieferung blockieren sollte
Für gruppenübergreifende Funktionen (--fail-on, --min-level, Sortierung) besitzt jeder Schweregrad zusätzlich ein internes Gewicht. So lassen sich Findings beider Skalen vergleichen.

7. Reportformate

ACI kennt vier Reportformate (--format console,json,html,sarif, kommagetrennt kombinierbar). Alle stellen die Prüfgruppen getrennt dar – jeweils mit eigener Schweregrad-Zusammenfassung.

Konsole

Farbiger Textreport, je Gruppe ein eigener Abschnitt. Mit --no-color ohne ANSI-Farben.

JSON

Maschinenlesbarer Report. Jedes Finding enthält group, severity, fingerprint und (bei Guideline-Findings) eine url zur Regelseite. Der Report führt zusätzlich scanner_config, ruleset.integrity und – bei genutzter Waiver-Datei – einen waivers-Block.

HTML

Eigenständige HTML-Seite mit Kennzahlen-Karten je Gruppe; die Findings sind über --html-group-by wahlweise nach Regel oder nach Datei gruppiert.

SARIF

SARIF-2.1.0-Report (aci_report_<name>.sarif) für GitHub-/GitLab-/Azure-DevOps-Code-Scanning. Der Regelkatalog ist je check_id:rule_ref geführt; gewaiverte Findings erscheinen als suppressions, der Ruleset-Hash als Run-property.

8. Gruppe Sicherheit

Fünf Checks, gesteuert über rules/oracle.json bzw. rules/postgresql.json:

IDCheckPrüft auf
ACI-NAMENamenskonventionen reservierte Worte als Bezeichner, überlange Namen
ACI-PKGUnerwünschte Packages sicherheitskritische Packages/Funktionen/Sprachen
ACI-OBFObfuskierter Code gewrappter, kodierter oder verschleierter Code
ACI-SQLISQL Injection dynamisches SQL mit String-Konkatenation
ACI-DDLDDL im Code GRANT/CREATE/ALTER/DROP … im Code

Details zu diesen Checks: jeder Check kann in der Regeldatei über enabled ein-/ausgeschaltet werden; Wortlisten, Pakete, DDL-Schlüsselworte und Muster sind dort frei pflegbar.

SQL-Injection-Erweiterungen (ACI 2.22.0): Der ACI-SQLI-Check erkennt zusätzlich (a) eine gezielte interprozedurale Injection – ein ungeprüfter Wert, der über einen Routinenaufruf in das dynamische SQL einer Hilfsroutine derselben Datei fließt (Regel-Ref ACI-SQLI-IP; über "interprocedural": false abschaltbar); (b) Oracle-SQLi über DBMS_XMLGEN.NEWCONTEXT/DBMS_XMLQUERY.NEWCONTEXT, die die übergebene Abfrage ausführen und die EXECUTE IMMEDIATE-Erkennung umgehen; sowie (c) die PostgreSQL-Session-/GUC-Quelle current_setting/set_config als typisierte, benutzerkontrollierbare Eingabe.

MITRE-ATT&CK-Angriffsindikatoren

Zusätzlich zu den fünf Checks prüft die Gruppe Sicherheit eine Reihe von Angriffsindikatoren nach MITRE ATT&CK (Oracle und PostgreSQL/EPAS). Es handelt sich um Code-Muster, die auf konkrete Angriffstechniken hindeuten – etwa datenbankweite LOGON-Trigger (Persistenz), das Löschen von Audit-Daten (Defense Evasion), den Zugriff auf interne Hash-Tabellen (Credential Access) oder ausgehende JDWP-Verbindungen (Command & Control). Jedes Finding ist mit MITRE-Taktik und -Technik (T…) versehen und verlinkt auf die zugehörige Seite bei attack.mitre.org.

Die Indikatoren ergänzen die fünf Checks gezielt um Muster, die diese noch nicht erfassen – es entstehen keine doppelten Findings. Sie liegen als eigene Regeldateien je Taktik unter rules/mitre/<dialect>/ (z.B. rules/mitre/oracle/ta0003_persistence.json) und sind dort frei erweiterbar.

TaktikBeispiel-Indikatoren
Execution (TA0002)DBMS_MLE, CREATE JAVA SOURCE
Persistence (TA0003)AFTER LOGON ON DATABASE, Scheduler-Job job_type=>'EXECUTABLE', CREATE OR REPLACE … SYS.…
Privilege Escalation (TA0004)AUTHID DEFINER, IDENTIFIED BY VALUES
Defense Evasion (TA0005)_oracle_script, NOAUDIT, DELETE FROM SYS.AUD$
Credential Access (TA0006)SYS.USER$, DBMS_SYSTEM.GET_ENV
Lateral Movement (TA0008)DBMS_FILE_TRANSFER
Exfiltration / C2 (TA0010/TA0011) DBMS_DEBUG_JDWP.CONNECT_TCP, hartkodierte Relay-URLs, DBMS_NETWORK_ACL_ADMIN
Impact (TA0040)CREATE LIBRARY, DBMS_ADVANCED_REWRITE
Defense Impairment (TA0112) DBMS_SQL_FIREWALL.DISABLE
Viele dieser Muster haben auch legitime DBA-Verwendung. Ein MITRE-Finding ist ein Indikator, der ein Review verdient – kein sicherer Nachweis eines Angriffs. Laufzeitbezogene ATT&CK-Techniken (Logins, Audit-Trail-Auswertung, DB-Zustand) kann ein statischer Scanner nicht erkennen; dafür sind die SOC-Regeln des Oracle-MITRE-Leitfadens vorgesehen.

9. Gruppe Coding Guidelines

Die Gruppe „Coding Guidelines“ setzt eine Teilmenge der Trivadis PL/SQL & SQL Coding Guidelines (Version 4.4, Apache-2.0-Lizenz) um. Die Regeln tragen ihre originalen IDs (G-NNNN) und verlinken auf die jeweilige Trivadis-Regelseite.

Abdeckung

ACI ist ein muster-/heuristikbasierter Scanner. Umgesetzt ist die Teilmenge der Guidelines, die sich damit zuverlässig prüfen lässt – rund 35 aktive Regeln. Regeln, die eine vollständige PL/SQL-Syntaxanalyse erfordern (z.B. unbenutzte Variablen, toter Code, Kontrollfluss), sind in den Regeldateien mit "enabled": false und "coverage": "needs-parser" hinterlegt. Sie erscheinen in --list-checks als bekannt, aber nicht geprüft.

Beispiele aktiver Regeln

IDRegelLevel
G-2180keine gequoteten Bezeichner Major
G-2320Datentyp VARCHAR nie verwenden Major
G-3145kein SELECT * Major
G-3310kein COMMIT in einer Schleife Critical
G-4310keine GOTO-Anweisungen Major
G-6010dynamisches SQL über Variable ausführenMajor
G-7125CREATE OR REPLACE verwenden Major
G-9010Formatmaske bei TO_DATE Major

Detektoren

Jede Guideline-Regel besitzt eine Detektor-Beschreibung – entweder ein regulärer Ausdruck (type: regex) oder ein eingebauter Detektor mit Sonderlogik (type: builtin, z.B. für Namenspräfixe, COMMIT-in-Schleife oder fehlende Parametermodi).

Die Coding Guidelines gelten ausschließlich für Oracle PL/SQL. Bei --dialect postgresql wird nur die Gruppe Sicherheit geprüft.

10. Regeldateien (JSON)

Alle Regeln liegen außerhalb des Programmcodes.

Sicherheit

Eine Datei je Dialekt (rules/oracle.json, rules/postgresql.json) mit den fünf Checks, Wortlisten, Paketen und DDL-Schlüsselworten.

Coding Guidelines

Eine Datei je Trivadis-Kategorie unter rules/guidelines/oracle/. Eine Regel hat folgende Form:

{
  "id": "G-3145",
  "title": "Kein SELECT * aus Tabelle oder View",
  "severity": "Major",
  "characteristics": "Maintainability, Reliability",
  "message": "Verwendung von SELECT * .",
  "recommendation": "Benötigte Spalten explizit aufzählen.",
  "url": "https://trivadis.github.io/.../g-3145/",
  "enabled": true,
  "detector": { "type": "regex", "target": "code",
                "pattern": "\\bSELECT\\s+\\*" }
}

Eine neue Regel ist durch einen weiteren Eintrag in der passenden Kategoriedatei aktivierbar; ein regex-Detektor benötigt keinen Code. target wählt, ob im Code (code/masked) oder im Inhalt von String-Literalen (string) gesucht wird.

Auf gültiges JSON achten; Rückwärtsstriche in regulären Ausdrücken JSON-konform verdoppeln (\\b). ACI prüft jede Regeldatei beim Laden und meldet Fehler verständlich.

11. CI/CD-Integration

Mit --fail-on liefert ACI einen auswertbaren Exit-Code. Der Schwellwert akzeptiert jeden Schweregrad beider Skalen; verglichen wird über das interne Gewicht.

Exit-CodeBedeutung
0kein Finding erreicht die Schwelle
1mindestens ein Finding erreicht/übersteigt die Schwelle
2Werkzeug-/Eingabefehler (ungültige Regeldatei, Pfad nicht gefunden, ungültige Option) oder – mit --strict-internal-errors bzw. --strict-waivers – ein interner Check-Fehler bzw. eine fehlerhafte Waiver-Datei
# Build schlägt bei sicherheitsrelevanten Findings fehl
python3 aci.py ./src --dialect oracle --group security \
        --format json --output-dir reports/ --fail-on high

CI/CD-Profile

Damit man in der Pipeline keine lange Schalter-Kette zusammenstellen muss, kennt ACI vordefinierte Profile. Ein Profil ist ein benanntes Bündel sinnvoller Voreinstellungen; es legt eine Vorgabe-Schicht zwischen aci.ini und die expliziten Kommandozeilen-Schalter. Ein ausdrücklich gesetzter Schalter hat also weiterhin Vorrang vor dem Profil.

ProfilZweckgesetzte Vorgaben
advisoryberatend – scannt und berichtet, blockiert nie--group security --fail-on none --format console,json,sarif --safe-report
ciempfohlener harter Gate – blockiert ab High--group security --fail-on high --format console,json,sarif --safe-report --strict-internal-errors
strictstrengster Gate – ci plus strenge Waiver-/Regelprüfung wie ci, zusätzlich --strict-waivers --require-trusted-rules
auditvollständige Prüfung für ein manuelles Review (mit Kontext, ohne Blockade) --group all --fail-on none --format console,html --taint-sources
# empfohlener Gate
python -m aci sql/ --profile ci

# Gate mit Waiver-Datei
python -m aci sql/ --profile ci --waivers aci-waivers.json

# einzelne Vorgabe des Profils ueberschreiben
python -m aci sql/ --profile ci --fail-on critical
Die Profile advisory, ci und strict setzen bereits --fail-on high (bzw. none bei advisory) und --safe-report - der folgende ausführliche Gate-Aufruf zeigt dieselben Schalter einzeln.

Baseline-/Diff-Modus (Legacy-Adoption)

Auf gewachsenem Code liefert der erste Scan oft zu viele Findings, um sofort ein hartes Gate zu schalten. Der Baseline-Modus schreibt den aktuellen Stand als akzeptierten Ausgangspunkt fest und meldet fortan nur noch neu hinzugekommene Findings. Die Bindung erfolgt über denselben inhaltsgebundenen Fingerabdruck wie bei den Waivern – unabhängig von absolutem Pfad und Report-Kontext.

# Einmalig: aktuellen Stand festschreiben (beendet mit Exit 0)
python -m aci sql/ --write-baseline aci-baseline.json

# In der Pipeline: nur neue Findings zaehlen fuers Gate
python -m aci sql/ --profile ci --baseline aci-baseline.json

ACI-INTERNAL (Werkzeugfehler) wird nie über die Baseline unterdrückt; eine fehlende oder fehlerhafte Baseline-Datei führt fail-closed zu Exit-Code 2.

Ab ACI 2.22.1: Die Baseline führt je Fingerabdruck eine Anzahl (Multiset, Format Version 2). Zwei identische Befunde in einer Datei teilen denselben Fingerabdruck; sind k Vorkommen bekannt und treten m auf, gelten min(k, m) als bekannt, die übrigen bleiben sichtbar. Legacy- Baselines aus 2.22.0 ({"fingerprints": […]}) bleiben lesbar; die Datei wird atomar geschrieben und streng validiert.

Inline-Suppression (-- aci:ignore)

Ergänzend zur zentralen Waiver-Datei lassen sich einzelne Fundstellen mit einem Kommentar direkt am Code stummschalten: -- aci:ignore (dieselbe Zeile) bzw. -- aci:ignore-next-line (nächste Zeile), optional auf Regeln beschränkt via [ACI-SQLI, …]. Unterdrückte Findings zählen nicht für --fail-on; ihre Anzahl wird als Hinweis gemeldet. Werkzeugfehler (ACI-INTERNAL) sind bewusst nicht unterdrückbar.

EXECUTE IMMEDIATE v_sql;  -- aci:ignore[ACI-SQLI] SEC-42: geprueft

Empfohlener CI/CD-Gate

Für eine Deployment-Pipeline empfiehlt sich der folgende Aufruf. Er blockiert den Build bei sicherheitsrelevanten Findings und erzeugt zugleich archivierbare, kontextfreie Report-Artefakte.

Audit-relevant – mindestens --fail-on high: Für eine echte Deployment-Blockade unsicherer Befehle muss die Schwelle auf --fail-on high gesetzt werden; --fail-on critical ist für ein verlässliches Sicherheits-Gate zu schwach. Viele CI-relevante administrative Befunde sind bewusst mit High bewertet, nicht mit Critical – etwa serverseitiger Dateizugriff, ausgehende Netzwerkverbindungen, dynamisches SQL aus ungeprüften Parametern sowie ALTER SYSTEM SET oder ein riskantes CREATE EXTENSION dblink. Ein PostgreSQL-Skript mit ALTER SYSTEM SET work_mem und CREATE EXTENSION dblink läuft mit --fail-on critical ungehindert durch, wird aber mit --fail-on high korrekt blockiert.
# PostgreSQL
python -m aci sql/ \
  --dialect postgresql \
  --group security \
  --fail-on high \
  --strict-internal-errors \
  --safe-report \
  --format console,json,sarif \
  --output-dir reports

# Oracle (analog)
python -m aci sql/ \
  --dialect oracle \
  --group security \
  --fail-on high \
  --strict-internal-errors \
  --safe-report \
  --format console,json,sarif \
  --output-dir reports

Die Schalter im Einzelnen:

  • --group security – nur die Sicherheits-Findings sind Gate-relevant; Coding-Guideline-Findings bleiben beratend und blockieren den Build nicht.
  • --fail-on high – Exit-Code 1 ab Schweregrad High, blockiert also High und Critical (siehe Audit-Hinweis oben).
  • --strict-internal-errors – schlägt ein Check intern fehl, endet der Lauf mit Exit-Code 2; eine stille Prüflücke fällt so nicht unbemerkt durch den Gate.
  • --safe-report – bündelt --no-context, --redact-secrets und --redact-paths: die Report-Artefakte enthalten keinen Quelltext-Kontext, keine erkannten Geheimnisse und keine absoluten Pfade und können gefahrlos als CI-Artefakt aufbewahrt werden.
  • --format console,json,sarif – Konsole für das Build-Log, JSON zur Archivierung/Auswertung, SARIF für die Integration in Code-Scanning (z.B. GitHub Code Scanning).
  • --output-dir reports – Zielverzeichnis der JSON-/SARIF-Artefakte.

Der Gate ist bestanden, wenn der Lauf mit Exit-Code 0 endet. Exit-Code 1 bedeutet mindestens ein Finding ab High, Exit-Code 2 einen Werkzeug- oder internen Prüffehler – beide sollten den Build stoppen.

Ausnahmeprozess / Waiver

Reale Pipelines brauchen einen kontrollierten Weg, einzelne Befunde zu akzeptieren – ohne --fail-on global abzuschwächen oder Regeln zu deaktivieren. Beides würde den Gate als Ganzes untergraben. Dafür kennt ACI Waiver: eine versionierte JSON-Datei mit einem Eintrag je akzeptierter Ausnahme. Ein gewaivertes Finding bleibt im Report sichtbar – deutlich als Waived markiert, mit Ticket, Owner, Ablaufdatum und Begründung –, zählt aber nicht mehr für --fail-on.

Bindung über den Fingerabdruck. Jedes Finding trägt einen inhaltsgebundenen Fingerabdruck: einen SHA-256-Hash aus Check-ID, Regelreferenz, Dateiname und dem (auf Leerzeichen normalisierten) beanstandeten Code – bewusst ohne Zeilennummer. Verschiebt sich der Code (neue Zeilen darüber), bleibt der Fingerabdruck gleich und der Waiver gilt weiter. Ändert sich der beanstandete Code selbst, ändert sich der Fingerabdruck: der Waiver verfällt automatisch und der Code wird neu geprüft. Ein Waiver deckt also diesen Code, nicht diese Stelle für immer. Der Fingerabdruck steht in jedem Report (Konsole, JSON, HTML, SARIF) und kann direkt in die Waiver-Datei übernommen werden.

Waiver-Datei (z. B. aci-waivers.json) – eine JSON-Liste; Pflichtfelder je Eintrag: fingerprint, ticket, owner, expires (YYYY-MM-DD), reason; optional created und risk_accepted:

[
  {
    "fingerprint": "39e7c9525757badd",
    "ticket": "SEC-1423",
    "owner": "ak@red-database-security.com",
    "expires": "2026-09-30",
    "reason": "Dynamisches SQL nur über validierte Whitelist-Konstante; Refactor in SEC-1423 geplant.",
    "created": "2026-05-24",
    "risk_accepted": true
  }
]

In der Pipeline wird die Datei über --waivers übergeben:

python -m aci sql/ --group security --fail-on high \
  --waivers aci-waivers.json --safe-report \
  --format console,json,sarif --output-dir reports

Lebenszyklus eines Waivers:

ZustandBedingungWirkung
gültig Ablaufdatum ≥ heute und Fingerabdruck trifft ein Finding Finding wird Waived: sichtbar im Report, zählt nicht für --fail-on; in SARIF als suppressions (kind: external)
abgelaufen Ablaufdatum < heute Waiver greift nicht mehr; das Finding zählt wieder voll. ACI warnt über abgelaufene Waiver
bald fällig noch gültig, Ablauf ≤ 14 Tage entfernt ACI weist im Report darauf hin (Wiedervorlage)
verwaist Fingerabdruck trifft kein Finding ACI meldet den Waiver als entfernbar (Code geändert oder Tippfehler)

Eine fehlerhafte Waiver-Datei (ungültiges JSON, fehlende Pflichtfelder, kaputtes Datum, doppelter Fingerabdruck) führt standardmäßig nur zu einer Warnung – ein Tippfehler blockiert so nicht die ganze Pipeline. Mit --strict-waivers endet der Lauf bei einer fehlerhaften Waiver-Datei mit Exit-Code 2.

Die Waiver-Datei gehört unter Versionskontrolle. Änderungen daran laufen über den normalen Pull-Request-Review – das ist die eigentliche Freigabe-Kontrolle. Das Pflicht-Ablaufdatum erzwingt eine Wiedervorlage, sodass keine „ewigen“ Ausnahmen entstehen.

Regelintegrität (Ruleset-Hash)

Die Regeldateien bestimmen, was ACI prüft. Über --rules, --rules-dir, --guidelines-dir und --mitre-dir lassen sie sich austauschen – flexibel, in einer CI/CD-Pipeline aber auch ein Manipulationsrisiko: Wer die Regeln verändert, kann den Gate schwächen. ACI macht den verwendeten Regelstand daher überprüfbar.

Jeder Report enthält einen Ruleset-Hash – einen SHA-256 über den Inhalt aller tatsächlich geladenen Regeldateien (Sicherheits-Regelsatz, Coding-Guidelines, MITRE-Indikatoren). Der Hash ist stabil und reproduzierbar; ein Team kann ihn in der Pipeline gegen einen erwarteten Wert prüfen. Zusätzlich weist ACI je Datei aus, ob sie vertrauenswürdig ist – also Teil der mit ACI ausgelieferten, gebündelten Regeln (innerhalb des installierten aci-Pakets) – oder aus einem benutzerdefinierten Pfad stammt (untrusted). Untrusted Regeldateien werden im Report sichtbar markiert und auf der Standard-Fehlerausgabe gewarnt.

Der Ruleset-Hash erscheint in allen Reportformaten: Konsole (Zeile Regeln), JSON (ruleset.integrity), HTML (Zeile Regelintegrität) und SARIF (runs[].properties.aci_ruleset_hash).

Mit --require-trusted-rules wird daraus ein harter Kontrollpunkt: ACI bricht mit Exit-Code 2 ab, sobald eine Regeldatei von außerhalb des Pakets geladen würde. Für ein abgesichertes CI-Gate:

python -m aci sql/ --group security --fail-on high \
  --require-trusted-rules --safe-report \
  --format console,json,sarif --output-dir reports

SARIF-Regelkatalog je Regelreferenz

Der SARIF-Regelkatalog (tool.driver.rules) wird je konkreter Regelreferenz geführt: die ruleId ist check_id:rule_ref – etwa ACI-PKG:utl_http oder ACI-OBF:CHR-CHAIN – und nicht nur check_id. So verfolgen GitHub- und GitLab-Security-Dashboards einzelne Regeln getrennt (z.B. jedes unerwünschte Paket unter ACI-PKG oder die einzelnen MITRE-Indikatoren). check_id und rule_ref stehen zusätzlich in den properties jedes Regeleintrags. Fehlt eine Regelreferenz oder gleicht sie der check_id, bleibt es bei der reinen check_id. Gewaiverte Findings erscheinen als suppressions, jeder Befund trägt seinen Fingerabdruck als partialFingerprints.

12. How to use in Enterprise

Dieser Abschnitt beschreibt, wie ACI im Unternehmen kontrolliert ausgerollt wird: ein gestufter Rollout, wiederverwendbare interne Runbooks mit konkreten Beispielen sowie eine Onboarding-Guidance für neue Entwickler und Teams. Alle Beispiele sind versionsfest für ACI 2.20.0.

12.1 Gestufter Rollout

ACI wird nicht sofort als harter Gate eingeführt, sondern in drei Stufen über die vordefinierten CI/CD-Profile (--profile). So gewöhnen sich Teams an die Befunde, bevor der Build blockiert.

StufeProfilWirkungZiel
1 · Beobachtenadvisory scannt und berichtet, blockiert nie Baseline erheben, Rauschen sichten, Waiver vorbereiten
2 · Gateci blockiert ab High (Gruppe Sicherheit) regulärer Schutz für Merge-/Build-Pipelines
3 · Strengstrict strengster Gate, strikte interne Fehler/Waiver regulierte/sicherheitskritische Bereiche

Empfehlung: mindestens zwei Wochen advisory fahren, die Findings als Baseline dokumentieren und akzeptierte Altlasten per Waiver abdecken, bevor auf ci umgestellt wird.

12.2 Runbook — Pipeline-Gate einrichten

Der empfohlene Gate-Aufruf blockiert sicherheitsrelevante Befunde, erzeugt ein archivierbares SARIF-Artefakt und gibt keine Geheimnisse oder absoluten Pfade preis. Dialekt explizit setzen – ohne --dialect postgresql gilt der Oracle-Regelsatz.

# Oracle
python -m aci sql/ \
    --dialect oracle \
    --profile ci \
    --fail-on high \
    --safe-report \
    --format sarif \
    --output-dir reports/

# PostgreSQL / EPAS
python -m aci sql/ \
    --dialect postgresql \
    --profile ci \
    --fail-on high \
    --safe-report \
    --format sarif \
    --output-dir reports/

--safe-report bündelt --no-context, --redact-secrets und --redact-paths – ideal für CI-Artefakte, die in Dashboards oder Tickets landen. Exit-Code 0 gibt frei, 1 blockiert (Gate), 2 signalisiert einen Werkzeug-/Eingabefehler.

12.3 Runbook — Regelintegrität & Versionsbindung

In größeren Organisationen sollte der verwendete Regelstand reproduzierbar und manipulationssicher sein. ACI weist einen Ruleset-Hash aus und kann ihn gegen einen Sollwert prüfen (fail-closed).

# 1) Sollstand einmalig festschreiben (Lock-Datei eingecheckt im Repo)
python -m aci --dialect oracle --print-effective-config > /dev/null
#    Ruleset-Hash aus dem Report (Block "ruleset_integrity") in
#    aci-ruleset.lock ({"ruleset_sha256": ""}) ablegen.

# 2) In der Pipeline gegen den Sollstand verifizieren
python -m aci sql/ \
    --dialect oracle --profile ci --fail-on high \
    --ruleset-lock aci-ruleset.lock \
    --require-trusted-rules \
    --format sarif --output-dir reports/
  • --ruleset-lock / --expected-ruleset-sha256: Abbruch mit Exit-Code 2 bei abweichendem Regelstand.
  • --require-trusted-rules: nur die gebündelten Regeln zulassen, keine lokal geänderten Regelpfade.
  • ACI-Version im Pipeline-Image pinnen (z.B. pip install aci==2.20.0), damit Regelsatz und Werkzeug zusammenpassen.

12.4 Runbook — kontrollierter Ausnahmeprozess (Waiver)

Einzelne akzeptierte Befunde werden über eine versionierte Waiver-Datei freigegeben – ohne den Gate global abzuschwächen oder Regeln zu deaktivieren. Jeder Eintrag trägt Fingerabdruck, Ticket, Owner, Ablaufdatum und Begründung.

python -m aci sql/ \
    --dialect postgresql --profile ci --fail-on high \
    --waivers aci-waivers.json \
    --strict-waivers \
    --format sarif --output-dir reports/

Gewaiverte Befunde bleiben im Report sichtbar (als suppressions im SARIF), zählen aber nicht für das Gate. Abgelaufene Waiver unterdrücken nicht mehr; ACI warnt über abgelaufene, bald fällige und verwaiste Einträge. Empfehlung: Waiver-Datei reviewpflichtig (Code-Owner) halten und Ablaufdaten setzen.

12.5 Onboarding-Guidance für neue Teams

Ein kurzer, wiederholbarer Einstieg senkt die Hürde und sorgt dafür, dass ACI lokal – vor dem Push – genutzt wird.

  1. Lokal installieren & smoke-testen: pip install aci==2.20.0, dann python -m aci pfad/zur_datei.sql --dialect <oracle|postgresql>.
  2. Vor dem Commit lokal scannen: denselben --profile ci --fail-on high-Aufruf wie die Pipeline verwenden, damit es im CI keine Überraschungen gibt.
  3. Reports lesen: Console für die schnelle Rückmeldung, HTML für Review-Meetings, SARIF für das Code-Scanning der Plattform. Jeder Befund nennt Regel-ID, Schweregrad und (sofern aktiv) Quelltext-Kontext.
  4. Befund einordnen: echtes Risiko → beheben; bewusste, geprüfte Ausnahme → Waiver mit Ticket/Owner/Ablauf; vermuteter Fehlalarm → mit --debug reproduzieren und an das ACI-/Security-Team melden.
  5. Grenzen kennen: ACI ist ein heuristischer Filter, kein Ersatz für manuelles Review (siehe 13. Grenzen & Hinweise).

Verantwortlichkeiten: ein zentrales Team pflegt Regelsatz, Lock-Datei und Profile; die Entwicklungsteams beheben Befunde bzw. beantragen Waiver; Security gibt Waiver und Ausnahmen frei.

12.6 Beispiel — Plattform-Integration

ACI ist plattformneutral: Exit-Codes und SARIF sind die universellen CI/CD-Schnittstellen. Beispiel für einen GitHub-Actions-Schritt mit Code-Scanning-Upload:

- name: ACI security gate
  run: |
    pip install aci==2.20.0
    python -m aci sql/ \
      --dialect postgresql --profile ci --fail-on high \
      --ruleset-lock aci-ruleset.lock --safe-report \
      --format sarif --output-dir reports/
- name: Upload SARIF
  if: always()
  uses: github/codeql-action/upload-sarif@v3
  with:
    sarif_file: reports/

Der if: always()-Schritt lädt das SARIF auch dann hoch, wenn der Gate blockiert – so bleiben die Befunde im Security-Dashboard sichtbar.

12.7 Rule-Observability — welche Regeln wie oft anschlagen und warum

Für ein gepflegtes Regelwerk lohnt es sich auszuwerten, welche Regeln wie häufig anschlagen – und warum. ACI liefert dafür keine eingebaute Statistik, aber der JSON-Report (--format json) ist dafür die stabile, maschinenlesbare Grundlage: Teams bauen darauf eine eigene Auswertung. Jeder Befund trägt u.a. check_id, rule_ref (konkrete Regelreferenz), group, severity, file, line und einen stabilen fingerprint.

So funktioniert Rule-Observability – in fünf Schritten:

  1. Reports erzeugen: ACI je Lauf mit --format json --output-dir reports/ ausführen (z.B. in jeder CI-Pipeline) und die Artefakte zentral ablegen – eines pro Lauf/Tag/Commit.
  2. Je Regel aggregieren: über .files[].findings[] iterieren und nach rule_ref zählen (das „wie oft“).
  3. Ursache verstehen: das „warum“ liefert der Befund selbst – message begründet die Regel, snippet/context zeigen die Fundstelle, rule_ref verweist auf die Regeldatei.
  4. Deuten & reagieren: Muster gegen die Tabelle unten abgleichen (echtes Risiko beheben, Fehlalarm schärfen/waiven, Lücken dokumentieren).
  5. Trends sichern: Lauf-Zeitpunkt (generated / runtime.started_at_utc) und Ruleset-Hash (ruleset_integrity) mitführen, damit Verläufe gegen einen bekannten Regelstand vergleichbar bleiben.

Reports zentral einsammeln (z.B. als CI-Artefakte pro Lauf/Tag) und je rule_ref aggregieren. Ad-hoc mit jq – Häufigkeit je Regelreferenz:

# Top-Regeln über alle Reports im Ordner, absteigend gezählt
jq -r '.files[].findings[].rule_ref' reports/*.json \
  | sort | uniq -c | sort -rn

Reproduzierbare Auswertung in Python – Häufigkeit je Regel inkl. Schweregrad und betroffener Dateien (rein aus Standardbibliothek):

import collections, glob, json

counts = collections.Counter()
sev    = {}
files  = collections.defaultdict(set)

for path in glob.glob("reports/*.json"):
    with open(path, encoding="utf-8") as fh:
        report = json.load(fh)
    for entry in report.get("files", []):
        for f in entry.get("findings", []):
            rid = f.get("rule_ref") or f.get("check_id")
            counts[rid] += 1
            sev[rid] = f.get("severity")
            files[rid].add(f.get("file"))

for rid, n in counts.most_common():
    print(f"{n:5d}  {sev.get(rid,''):8s}  {rid}  "
          f"({len(files[rid])} Datei(en))")

Das „warum“ liefert der Befund selbst: message begründet die Regel, snippet/context zeigen die Fundstelle (sofern nicht per --no-context/--safe-report unterdrückt), und rule_ref verweist auf die konkrete Regeldatei. Für Trends über die Zeit den Lauf-Zeitpunkt aus generated bzw. runtime.started_at_utc mitführen und je Tag/Commit ablegen.

BeobachtungMögliche DeutungReaktion
Eine Regel dominiert die Treffer systemisches Muster im Code oder zu breite Regel (Falsch-Positiv-Quelle) Code-Pattern beheben; bei FP Regel schärfen oder gezielt Waiver setzen
Regel schlägt nie an im Bestand nicht relevant oder Detektor greift nicht Abdeckung prüfen (Beispiel scannen); ggf. dokumentieren
Treffer auf wenige Dateien konzentriert lokaler Hotspot / Alt-Komponente gezieltes Review, ggf. Refactoring priorisieren
Plötzlicher Anstieg nach einem Commit neue Risikoklasse eingeführt im Merge-Review adressieren, Gate greift bereits

Hinweis: Zählungen sind über den fingerprint dedupliziert vergleichbar – derselbe Befund behält seinen Fingerabdruck über Läufe hinweg, solange sich der beanstandete Code nicht ändert. Den Ruleset-Hash (ruleset_integrity) mitprotokollieren, damit Trends gegen einen bekannten Regelstand interpretierbar bleiben.

13. Grenzen & Hinweise

ACI ist ein heuristischer statischer Review-Assistent. Er ersetzt kein vollständiges Code-Review, keinen formalen Sicherheitsbeweis und keine Datenbank-Härtung.

Analysearchitektur

ACI verwendet eine geschichtete Analysepipeline:

  1. Source Loading, Größenprüfung und Ausschlussregeln
  2. Lexikalische Analyse und Maskierung von Kommentaren/String-Literalen
  3. Leichtgewichtige Parser-/IR-Extraktion für Routinen, Statements, Assignments und dynamisches SQL
  4. Regelprüfungen mit heuristischer und IR-gestützter Analyse
  5. Reporting in Console, JSON, HTML oder SARIF

Die Parser-/IR-Schicht reduziert typische False Positives, indem sie Statement-Grenzen, Routine-Grenzen und die Reihenfolge von Assignments bewahrt. Sie wird unter anderem für dynamische DDL-Erkennung und SQL-Injection-Klassifikation genutzt.

Bekannte Grenzen der Parser-/IR-Schicht

  • kein vollständiger Oracle-PL/SQL- oder PostgreSQL-PL/pgSQL-Compiler,
  • keine vollständige Kontrollflussanalyse,
  • keine vollständige Datenflussanalyse über Funktionsaufrufe hinweg,
  • keine semantische Validierung gegen eine laufende Datenbank,
  • begrenztes Verständnis komplex generierter SQL-Fragmente,
  • False Positives und False Negatives bleiben möglich.

False Negatives bei generiertem und dynamischem SQL

ACI ist eine statische Analyse des übergebenen Quelltextes. Sie ist stark bei Code, der ihr vollständig vorliegt – sie sieht jedoch grundsätzlich nicht, was erst zur Build- oder Laufzeit entsteht. Dadurch können Befunde unentdeckt bleiben (False Negatives), insbesondere bei:

  • dynamischem SQL über mehrere Dateien hinweg – ein SQL-String wird in einer Datei bzw. Routine zusammengesetzt und in einer anderen ausgeführt; ACIs Datenflussanalyse ist routine-lokal und folgt Aufrufen nicht;
  • Build-Templates und Code-Generatoren – wird der eigentliche SQL-Code erst durch ein Template-/Makro-System, einen Präprozessor oder einen Generator erzeugt, prüft ACI nur die Vorlage, nicht das erzeugte Ergebnis;
  • Migrations-Generatoren – Werkzeuge, die SQL erst zur Laufzeit aus einem Modell ableiten (ORM-Migrationen, Schema-Diff-Tools), liefern ACI keinen statisch prüfbaren Quelltext;
  • konfigurations- und datengetriebenem SQL – SQL-Fragmente aus Konfigurationsdateien, Datenbankinhalten oder externen Eingaben, die erst zur Laufzeit zu einer Anweisung zusammengesetzt werden, liegen außerhalb des analysierten Textes.

Wo SQL erst nach dem von ACI geprüften Stand entsteht, sollte das erzeugte SQL zusätzlich geprüft werden – etwa indem der Generator- bzw. Migrations-Output selbst durch ACI läuft – und durch manuelles Review ergänzt werden. ACI ist ein Review-Assistent und CI-Signal, kein vollständiger Nachweis der Abwesenheit von SQL-Injection.

Reportpfade und Vertraulichkeit

JSON-, HTML- und SARIF-Reports können Dateipfade, Scan-Metadaten und Quelltext-Kontext enthalten. Absolute Pfade können interne Workspace-, Repository- oder CI-Runner-Strukturen offenlegen. Behandle Reports daher als interne Artefakte, sofern sie nicht geprüft oder bereinigt wurden.

Für externe Weitergabe sollten Reports ohne Quelltext-Kontext erzeugt werden (--no-context). Wird Kontext benötigt, sollte --redact-secrets oder --safe-report verwendet werden. Die Redaction ist heuristisch und bietet keine Vollständigkeitsgarantie.

Empfehlung: ACI als ersten, automatisierten Filter in den Entwicklungs- und Freigabeprozess einbinden und die Findings als Ausgangspunkt für ein gezieltes manuelles Review verwenden.

14. Debugging

Dieser Abschnitt fasst zusammen, wie sich ein ACI-Lauf im Fehlerfall eingrenzen lässt – sowohl bei einem Werkzeugfehler als auch bei einem fragwürdigen Finding.

Erste Schritte

OptionZweck
--debugzeigt bei einem unerwarteten Fehler den vollständigen Python-Traceback statt nur einer Kurzmeldung.
--dump-irgibt die Parser-/IR-Sicht einer einzelnen Datei als JSON aus und beendet sich. Hier sieht man, wie ACI Routinen, Statements und dynamisches SQL erkannt hat – die häufigste Ursache für ein unerwartetes Finding.
--list-checkszeigt, welche Checks und Regeln für Dialekt und Gruppe aktiv sind. Fehlt ein erwartetes Finding, ist oft die zugehörige Regel inaktiv oder der Dialekt falsch.

Exit-Codes deuten

0 = Gate bestanden, 1 = --fail-on-Schwelle erreicht, 2 = Werkzeug-/Eingabefehler oder ein durch --strict-* erzwungener Abbruch. Bei 2 verrät die Meldung auf der Standard-Fehlerausgabe (FEHLER: …) die Ursache.

Interne Fehler (ACI-INTERNAL)

Schlägt ein Check intern fehl, stoppt das den Scan nicht: der Fehler erscheint als Finding der Gruppe Interner Fehler (ACI-INTERNAL, Schweregrad High). Das ist ein Werkzeugfehler, kein Code-Mangel. Mit --debug lässt sich der Auslöser reproduzieren; ein --dump-ir der betroffenen Datei grenzt ihn weiter ein.

Fragwürdiges Finding eingrenzen

Bei einem vermuteten False Positive: die betroffene Anweisung in eine kleine, isolierte .sql-Datei kopieren und erneut scannen. Bestätigt sich der Fehlbefund, zeigt --dump-ir, ob bereits die IR die Anweisung falsch abgrenzt. Ein akzeptierter Einzelfall lässt sich über einen Waiver dokumentieren, statt das Gate abzuschwächen.

Reproduzierbarkeit prüfen

Weicht ein Lauf vom erwarteten Ergebnis ab, sollten zuerst der Ruleset-Hash und der Vertrauensstatus (Konsole, Zeile Regeln) verglichen werden – ein abweichender Hash bedeutet einen veränderten Regelstand. python3 -m compileall -q aci deckt Syntaxfehler nach lokalen Änderungen auf.

15. Glossar

Kurzdefinitionen der in dieser Dokumentation und im ACI-Code verwendeten Fachbegriffe.

BegriffBedeutung
CheckEine Prüfeinheit im Code (z.B. SqlInjectionCheck). Ein Check wertet eine oder mehrere Regeln aus.
RegelEin in JSON beschriebenes Prüfmuster (aci/rules/) – Wortliste, Regex oder Builtin-Detektor. Regeln sind ohne Code-Änderung erweiterbar.
DetectorDer Mechanismus, mit dem eine Regel matcht: regex (Muster) oder builtin (benannte Funktion in aci/checks/detectors.py).
FindingEin einzelner Befund: Regel, Schweregrad, Datei, Zeile/Spalte, Meldung, ggf. Codeausschnitt und Empfehlung.
Schweregrad / GewichtStufe eines Findings (Warning, High, Critical bzw. Info … Blocker). Jedem Grad liegt ein numerisches Gewicht zugrunde, das den Vergleich über beide Skalen erlaubt (für --fail-on).
IR (Intermediate Representation)Leichtgewichtige Zwischenform des Codes: Routinen, Statements, Zuweisungen, dynamisches SQL. Kein vollständiger SQL-Compiler.
MaskierungAufbereitete Codefassung (code_masked), in der Kommentar- und String-Inhalte ersetzt sind, damit Detektoren nicht auf Daten statt Code anschlagen.
Taint„Verunreinigung“ – ein Wert, der aus einer ungeprüften Quelle (Benutzereingabe, Parameter, Tabelle) stammt und nicht als sicher gelten kann.
SinkDie Stelle, an der ein getainteter Wert gefährlich wird – bei SQL-Injection das EXECUTE IMMEDIATE des dynamischen SQL.
Taint-QuelleDie Herkunft des getainteten Werts: die aufbauende Zuweisung bzw. der Routinenkopf eines ungeprüften Parameters. ACI weist sie als zusätzliche Fundstelle aus.
1st-/2nd-order SQL-Injection1st-order: dynamisches SQL aus einem ungeprüften Routine-Parameter. 2nd-order: aus einem zuvor aus Tabelle/Cursor gelesenen Wert.
FingerprintInhaltsgebundener SHA-256-Hash eines Findings (Check-ID, Regelreferenz, Dateiname, normalisierter Code) – ohne Zeilennummer. Bindet einen Waiver an den Befund.
WaiverEine kontrollierte, dokumentierte Ausnahme (Ticket, Owner, Ablaufdatum, Begründung). Ein gewaivertes Finding bleibt sichtbar, zählt aber nicht für --fail-on.
Ruleset-HashSHA-256 über den Inhalt aller geladenen Regeldateien – macht den verwendeten Regelstand überprüfbar.
SARIFStatic Analysis Results Interchange Format (Version 2.1.0) – ein standardisiertes JSON-Format für Static-Analysis-Ergebnisse, das u.a. GitHub-, GitLab- und Azure-DevOps-Code-Scanning einlesen.
SuppressionSARIF-Markierung eines unterdrückten Ergebnisses. ACI exportiert gewaiverte Findings als suppressions, sodass sie auditierbar bleiben.
MITRE ATT&CKÖffentliche Wissensbasis für Angreifer-Taktiken und -Techniken. ACI ordnet sicherheitskritische Muster passenden ATT&CK-Techniken zu.
GuidelineCoding-Guideline-Regel (Oracle nach Trivadis, PostgreSQL ACI-eigen) – beratend, kein Sicherheitsbefund.
False Positive / NegativeFehlbefund: ein gemeldetes, aber harmloses Muster (Positive) bzw. ein übersehenes, tatsächlich problematisches Muster (Negative).
